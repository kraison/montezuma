Extensions to Porter stemming yielding valid stems

Roy Anderson
2016

ABSTRACT

This document details developments to Porter Stemmer 1.01 (Common Lisp encoding) referred to in the following as Porter1, Porter2 and Porter3 which aim to increase the success rate of stemming by suffix removal. Porter2 rewrites some Porter1 suffix rules and adds a new kind of rule based on word exceptions. Porter3 uses Porter1 suffix removal but adds a new set of rules which work in conjunction with a word validator. The stemming outcomes using a sample of the Porter2 test file are compared in a parallel. In Porter3, a valid stem is a member of the set of validated words as determined by a validation function. The original entry point to the stemmer (stem) continues to give the Porter1 results (the shortest word forms) according to stemming rules but without reference to valid words. This is handled in Porter2 by encoding exceptions where the whole input word is tested possibly to bypass overly aggressive and blind stemming rules. But Porter3 introduces a new entry point (stem-to-valid-word) which has a different contract: to find the first valid stem as determined by a supplied validation function. This change was motivated by the need to support general Linguistics applications that need true stems (base forms) while honouring the existing behaviour for Information Retrieval applications based on the Porter1 Stemming Algorithm. The new version (Porter3) has been tested using the original test files (Porter1 and Porter2 sets of words (see below). Using the validate word function included in Porter3 there are a minority of unrecognised words but most words yield are either valid stems present in the set of valid words or which are stemmed to a valid word. When compared in parallel to Porter1, Porter3 not only satisfies the need to produce valid stems, but it also satisfies the goal of bringing together inflected word forms to a single base form.

INTRODUCTION

Porter1 reduces groups of terms by condensing them into a single string, even if the string is not a valid word (as would be determined by a dictionary lookup). For example, removal of the suffixes -ED, -ING, -ION, IONS results in the single term CONNECT which happens to be a valid word. On the other hand, removal of the suffixes -E, -ED, -EMENT, -EMENTS from inflected forms of ABATE results in the invalid word: ABAT. This is not an error but rather a feature of Porter Stemmer 1.x! Suffix stripping is intended to reduce the total number of terms in the information retrieval system, and hence reduces the size and complexity of the data in the system. Again, the purpose of suffix stripping is to reduce the number of word forms in a consistent manner.

Porter Stemmer 3.0, in contrast to the original algorithm, introduces stem validation while retaining the established stem stripping capacity that has made the Porter Stemmer so popular over the past decades. In the Porter Stemmer 3.0 algorithm, the stemming process is negotiated by calls to a word validator which can be either supplied by the stemmer, or passed amongst the arguments to the stem routine.

Stem Validation

Stem validation profoundly affects the behaviour of the Stemmer. With validation, (Porter3) output is guaranteed to be either a valid word, or NIL. A NIL return value from the validation test does not necessarily mean that a word is a misspelt, but this is a definite possibility. Alternatively, the stem form may simply be absent from the set of valid words. The following comparison of Porter1 to Porter3 using the first 4000 Porter2 diffs (two columns, the first is the input word and the second is the Porter2 output) demonstrates the strengths and weaknesses of the three interpretations of stemming. The number of sample words (4000) was chosen as a compromise between maximizing the sample size and retaining responsiveness of the Javascript routines providing table paging and column filtering. Note that Porter1 and Porter2 always return either a stripped word or the input word if no rules can be applied. On the other hand, Porter3 may return a NIL if the stemming rules do not lead to valid word. In addition, Porter3 may short circuit application of the stemming rules if the validation test returns an early non-NIL result. In the comparison of Porter1-3, the following values may appear in the Stemming Outcome Analyses columns:

  • absent

    The Stemmer returned NIL because the word is not a valid stem and the stemming rules failed to yield a valid word. The word may be incorrectly spelt or it could be a proper noun that is not included in the list of valid words.

  • invalid

    The Stemmer returned a word but it's not in the set of valid Porter3 words. Note that Porter3 returns a NIL value under this condition. It is most likely that this would be handled by the caller of the Stemmer by using the input word without stemming.

  • irregular

    The Stemmer identified a word in the finite list of inflected pairs of irregular word forms. The irregular words do not conform to the usual inflection patterns (e.g. thought/think, or sink/sank) so Porter3 bypasses suffix removal and returns the associated base form.

  • non-alphabetic

    The Stemmer's test for non-alphabetic characters excludes words with any digits, spaces or punctuation.

  • roman-numerals

    This is a new outcome in Porter3 added as an example of a set of words that should not be stemmed. The code was derived from a couple of routines found at www.rosettacode.org. The rosetta code was based on the assumption that the argument is always a well formed string of Roman numerals. It has been modified to return NIL if there are any invalid numerals or if the number is not well formed which is determined by encoding the number back to Roman numerals and comparing it to the input.

  • present

    No stemming rules were applied because the word was considered to be valid already.

  • stemmed

    The Stemmer successfully generated a valid stem word which is distinct from the input word.

  • stopword

    A stopword is a member of the most common words but which also do not convey a great deal of meaning. Porter3 identifies stopwords in a set of words that are not stemmed.

  • undersize

    No stemming was attempted because the word is too short.

The analysis of Porter1 to Porter3 stemming outcomes over the test sample of 4000 words highlights important differences between the approaches: Porter1 always returns a string, even if it does not conform to (Porter3) orthographical standards. Porter2 slightly improves the output quality reducing invalid outcomes and increasing the number of successfully stemmed words. Porter3 always returns NIL or a valid word as determined by the validator. If the validator represents membership of a rich source of words it will tend to be too generous when validating inflected word forms. This is evident in the limited sample of results in the Stemming Outcomes: many inflected forms are treated as valid even though they are not truly a stemmed base form (i.e. the desired uninflected word form). One solution for this would be to remove non-base forms from those that should be in the set of of stem forms. However, this is a major task there being over 100,000 word forms in a typical list such as the GCIDE list of entries.

Implementation Details

The following source code is from the Common Lisp implementation of Porter3. It is provided under the same open source licencing terms and conditions as Porter1. Porter3 is running under LispWorks 6.1 and Allegro Common Lisp 10.0.

Word Validators

At the core of Porter3 is a word validator implemented using a hash table of word symbols. During development using LispWorks the hash-table test was #'string-equal but this is not portable (to other Lisp machines, Allegro Common Lisp, for example) and the table test was changed to use #'eq which tests its two arguments for identical objects. This neccessated a word normalizing function since the eq function does not test for string equality. The normalize method returns a word in the form of a standardized Lisp symbol after conversion to uppercase. The valid word forms table is only loaded if it is needed (i.e. when the first call is made to the validator). The dynamic variable *porter-source-file* is assigned to the path name to the Porter3 source file location to obtain the location of the word lists (which are assumed to be in the same directory as the source file).

(eval-when (:compile-toplevel :load-toplevel :execute)
  (defvar *porter-source-file* (load-time-value (or #.*compile-file-pathname* *load-pathname*)))
  (defvar *porter-data-directory* (merge-pathnames "data/" *porter-source-file*)))

(defvar words (make-hash-table :test #'eq))

(defgeneric normalize (word) 
  "Return a symbol to be used to test whether WORD is valid.")

(defmethod normalize ((word string))
  (intern (string-upcase word)))

(defmethod normalize ((word symbol))
  word)

(defun read-valid-words (&optional (path (merge-pathnames 
                                          "porter-stemmer-words.txt"
                                          *porter-data-directory*)))
  ;; load in the valid words from the data directory, and irregular forms, and stopwords.
  (clrhash words)
  (with-open-file (wordsin path)
    (loop
      for line = (read-line wordsin nil nil)
      while line do
      (setf (gethash (normalize line) words) t))
    (load-irregulars)
    (load-stopwords)
    words))

(defun is-a-word (str)
  ;; Load the valid words unless they have already been loaded. Test if word is valid.
  (if (zerop (hash-table-count words))
      (read-valid-words))
  (gethash (normalize str) words))

Porter1 Compatiblity

The core function of the Porter Stemmer has been modified so that while returning exactly the same results as Porter1 (if the original entry point is called), it can also restrict results using a validation function. The new main entry point into the stemmer is stem-to-valid-word which expects to be given a word to check and which may be supplied with a validation function and a qualifier: exclusive that precludes the input word from being treated as a valid stem; even if it is a valid word. This functionality was used to build the current list of valid words. As you can see from the Common Lisp below, if a validate function is not passed to stem-to-valid-word, the default validator is used. Porter1 output is enabled if the stemmer is passed an explicit value of NIL for the validator.

(defun final-step (str validate)

        (return-from final-step str))

       ((ends str "est")           (setto str "")) ; wouldest, bravest

       ((ends str "men")           (setto str "man")) ; tradesmen

;; STEM is intended to provide Porter Stemming with the same results as Porter Stemmer 1.01 
(defun stem (str)

(defmethod stem-to-valid-word ((str string) &key (validate #'is-a-word) (exclusive t))
  (let ((len (length str))
        (inflected-word (string-downcase str)))
    ;; Strings of length < 3 don't go through the stemming process, 
    ;; although no mention is made of this in the published algorithm.
    ;; Remove the line to match the published algorithm.
    ;;
    ;; VALIDATE is an optional function that should return a non-NIL result
    ;; if the argument is valid.
    ;;
    ;; If EXCLUSIVE is True, STR is not a valid result, even if it is a
    ;; valid stem for other words.
    
    (block nil
      (if (find-if-not #'alpha-char-p str)
          (return (values str "non-alphabetic")))
      (if (<= len 2)
          (return (values str "undersize"))) ; /*-DEPARTURE-*/
      (setf str (make-array len 
                            :element-type 'character
                            :adjustable t
                            :fill-pointer len 
                            :initial-contents inflected-word))
      (cond
       (validate
        ;; Enable stemming to the first valid word
        (let ((validator (if exclusive 
                             #'(lambda(word) (and (string/= word inflected-word) 
                                                  (funcall validate word)))
                           validate)))
          (if (or 
               (funcall validator str)
               (funcall validator (step1ab str))
               (funcall validator (step1c str))
               (funcall validator (step2 str))
               (funcall validator (step3 str))
               (funcall validator (step4 str))
               (funcall validator (step5 str))
               (funcall validator (final-step str validator)))
              (values str                      
                      (if (string-equal str inflected-word)
                          "present"
                        "stemmed"))
            (values nil "absent"))))
       (t
        ;; Porter Stemmer 1.01: execute the stemming rules to get the shortest stem
        (step1ab str)
        (step1c str)
        (step2 str)
        (step3 str)
        (step4 str)
        (step5 str)
        str)))))

One more step

Preparing the Stemmer

The best results from Stemmer 3.0 would be obtained from a pure set of valid stems, preferably from a reliable source that maintains the list. Such a set of stems would maximize the number of correctly stemmed words and minimize the number of present and absent words. In the absence of a good list of word stems, the following code was used to reduce the GCIDE list of entries by about 30,000 words to a final word count of 93231. This means a significant improvement in stemming even if some stems such as "ace" must be reinstated if they are not to be stemmed to "ac".

The word reduction routine (reduce-words) takes a filename and loads its contents into the table of words (as it would for normal stemming). Each word is stemmed to obtain a word that is a candidate for a corresponding stem or it is assumed to be a stem form itself. Reduction continues until there are no changes in the size of the table of stems after stemming all the table entries in a complete iteration. If there is a change, the previous table is cleared out and the new table is put into place ready for another pass of stemming. When the table of stems stabilises, the final entries are collected and the sorted list of stems gets written out to a new file.


(defun stem-valid-word (word) 
  "If word can be stemmed return the stem, otherwise return the input word."
  (or (stem-to-valid-word word :validate #'is-a-word :exclusive t)
      word))

(defun write-words (wordtable name)    
  (let ((word-list (loop for word being the hash-keys of wordtable collect word)))
    (with-open-file (out (merge-pathnames name *PORTER-SOURCE-FILE*) 
                         :direction :output
                         :if-exists :supersede)
      (dolist (word (sort word-list #'string<))
        (format out "~a~%" word)))
    (format t "Wrote ~d words to ~a~%" (hash-table-count wordtable) name)))


(defun reduce-words ()
  "This is a maintenance routine that reduces the valid words to those that can't be stemmed."
  (let ((wordtable (make-hash-table :test #'eq)))
    (read-valid-words (merge-pathnames "porter-stemmer-words-gcide.txt" *porter-source-file*))
    (let ((pass 0))
      (loop
        (format t "Pass ~d vocabulary: ~d words~%" (incf pass) (hash-table-count words))
        (maphash
         #'(lambda(word value) (declare (ignore value))
             (setf (gethash (normalize (stem-valid-word word)) wordtable) :word))
         words)
        (when (= (hash-table-count wordtable) (hash-table-count words)) (return))
        (clrhash words)
        (setf words wordtable)
        (write-words wordtable (format nil "words-reduced-pass-~d.txt" pass))
        (setf wordtable (make-hash-table :test #'eq)))
      (format t "Completed vocabulary reduction: ~d words~%" (hash-table-count wordtable)))))

Perhaps the best criterion for removing suffixes from two words W1 and W2 to produce a single stem S, is to say that we do so if there appears to be no difference between the two statements `a document is about W1' and `a document is about W2'. So if W1=`CONNECTION' and W2=`CONNECTIONS' it seems very reasonable to conflate them to a single stem. But if W1=`RELATE' and W2=`RELATIVITY' it seems perhaps unreasonable, especially if the document collection is concerned with theoretical physics. (It should perhaps be added that RELATE and RELATIVITY are conflated together in the algorithm described here.) Between these two extremes there is a continuum of different cases, and given two terms W1 and W2, there will be some variation in opinion as to whether they should be conflated, just as there is with deciding the relevance of some document to a query. The evaluation of the worth of a suffix stripping system is correspondingly difficult.

The second point is that with the approach adopted here, i.e. the use of a suffix list with various rules, the success rate for the suffix stripping will be significantly less than 100% irrespective of how the process is evaluated. For example, if SAND and SANDER get conflated, so most probably will WAND and WANDER. The error here is that the -ER of WANDER has been treated as a suffix when in fact it is part of the stem. Equally, a suffix may completely alter the meaning of a word, in which case its removal is unhelpful. PROBE and PROBATE for example, have quite distinct meanings in modern English. (In fact these would not be conflated in our present algorithm.) There comes a stage in the development of a suffix stripping program where the addition of more rules to increase the performance in one area of the vocabulary causes an equal degradation of performance elsewhere. Unless this phenomenon is noticed in time, it is very easy for the program to become much more complex than is really necessary. It is also easy to give undue emphasis to cases which appear to be important, but which turn ut to be rather rare. For example, cases in which the root of a word changes with the addition of a suffix, as in DECEIVE/DECEPTION, RESUME/RESUMPTION, INDEX/INDICES occur much more rarely in real vocabularies than one might at first suppose. In view of the error rate that must in any case be expected, it did not seem worthwhile to try and cope with these cases.

It is not obvious that the simplicity of the present program is any demerit. In a test on the well-known Cranfield 200 collection7 it gave an improvement in retrieval performance when compared with a very much more elaborate program which has been in use in IR research in Cambridge since 1971(2,6). 2. THE ALGORITHM

To present the suffix stripping algorithm in its entirety we will need a few difinitions.

A consonant in a word is a letter other than A, E, I, O or U, and other than Y preceded by a consonant. (The fact that the term `consonant' is defined to some extent in terms of itself does not make it ambiguous.) So in TOY the consonants are T and Y, and in SYZYGY they are S, Z and G. If a letter is not a consonant it is a vowel.

A consonant will be denoted by c, a vowel by v. A list ccc... of length greater than 0 will be denoted by C, and a list vvv... of length greater than 0 will be denoted by V. Any word, or part of a word, therefore has one of the four forms:

    CVCV ... C
    CVCV ... V
    VCVC ... C
    VCVC ... V

These may all be represented by the single form

    [C]VCVC ... [V]

where the square brackets denote arbitrary presence of their contents. Using (VC){m} to denote VC repeated m times, this may again be written as

    [C](VC){m}[V].

m will be called the measure of any word or word part when represented in this form. The case m = 0 covers the null word. Here are some examples:

    m=0    TR,  EE,  TREE,  Y,  BY.
    m=1    TROUBLE,  OATS,  TREES,  IVY.
    m=2    TROUBLES,  PRIVATE,  OATEN,  ORRERY.

The rules for removing a suffix will be given in the form

    (condition) S1 -> S2

This means that if a word ends with the suffix S1, and the stem before S1 satisfies the given condition, S1 is replaced by S2. The condition is usually given in terms of m, e.g.

    (m > 1) EMENT ->

Here S1 is `EMENT' and S2 is null. This would map REPLACEMENT to REPLAC, since REPLAC is a word part for which m = 2.

The `condition' part may also contain the following:

*S  - the stem ends with S (and similarly for the other letters).

*v* - the stem contains a vowel.

*d  - the stem ends with a double consonant (e.g. -TT, -SS).

*o  - the stem ends cvc, where the second c is not W, X or Y (e.g.
       -WIL, -HOP).

And the condition part may also contain expressions with and, or and not, so that 

    (m>1 and (*S or *T))

tests for a stem with m>1 ending in S or T, while

    (*d and not (*L or *S or *Z))

tests for a stem ending witha double consonant other than L, S or Z. Elaborate conditions like this are required only rarely.

In a set of rules written beneath each other, only one is obeyed, and this will be the one with the longest matching S1 for the given word. For example, with

    SSES -> SS
    IES  -> I
    SS   -> SS
    S    ->

(here the conditions are all null) CARESSES maps to CARESS since SSES is the longest match for S1. Equally CARESS maps to CARESS (S1=`SS') and CARES to CARE (S1=`S').

In the rules below, examples of their application, successful or otherwise, are given on the right in lower case. The algorithm now follows:

    IES  -> I                          ponies    ->  poni
                                       ties      ->  ti
    SS   -> SS                         caress    ->  caress

                                    tann(ed)     ->  tan
                                    fall(ing)    ->  fall
                                    hiss(ing)    ->  hiss

                                    fil(ing)     ->  file

                                    sky          ->  sky

The algorithm is careful not to remove a suffix when the stem is too short, the length of the stem being given by its measure, m. There is no linguistic basis for this approach. It was merely observed that m could be used quite effectively to help decide whether or not it was wise to take off a suffix. For example, in the following two lists:

                  list A        list B
                  ------        ------
                  RELATE        DERIVATE
                  PROBATE       ACTIVATE
                  CONFLATE      DEMONSTRATE
                  PIRATE        NECESSITATE
                  PRELATE       RENOVATE

-ATE is removed from the list B words, but not from the list A words. This means that the pairs DERIVATE/DERIVE, ACTIVATE/ACTIVE, DEMONSTRATE/DEMONS- TRABLE, NECESSITATE/NECESSITOUS, will conflate together. The fact that no attempt is made to identify prefixes can make the results look rather inconsistent. Thus PRELATE does not lose the -ATE, but ARCHPRELATE becomes ARCHPREL. In practice this does not matter too much, because the presence of the prefix decreases the probability of an erroneous conflation.

Complex suffixes are removed bit by bit in the different steps. Thus GENERALIZATIONS is stripped to GENERALIZATION (Step 1), then to GENERALIZE (Step 2), then to GENERAL (Step 3), and then to GENER (Step 4). OSCILLATORS is stripped to OSCILLATOR (Step 1), then to OSCILLATE (Step 2), then to OSCILL (Step 4), and then to OSCIL (Step 5). In a vocabulary of 10,000 words, the reduction in size of the stem was distributed among the steps as follows:

    Suffix stripping of a vocabulary of 10,000 words
    ------------------------------------------------
    Number of words reduced in step 1:   3597
                  "                 2:    766
                  "                 3:    327
                  "                 4:   2424
                  "                 5:   1373
    Number of words not reduced:         3650

The resulting vocabulary of stems contained 6370 distinct entries. Thus the suffix stripping process reduced the size of the vocabulary by about one third.
